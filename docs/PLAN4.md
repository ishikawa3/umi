# かいしょう — 第4期計画（うみの陰画・業務システム版 / three.js）

PLAN.md（第1期）・PLAN2.md（第2期）・PLAN3.md（第3期）は「うみ」＝海しるAPIの
**データアート集**（フレームワーク無し・Canvas 2D・詩的・眺めるだけ）だった。

本計画は、まったく同じ海しるデータを使いながら**真逆の意匠**で見せる**別サイト**を作る。
名は仮に **かいしょう / VTS Console**（VTS = Vessel Traffic Service、船舶通航業務）。
うみが「鑑賞」なら、こちらは「監視・運用」。密なパネル、データグリッド、ステータスバー、
警報ティッカー、レイヤトグル。描画は **three.js / WebGL** の3D海図。ただし見た目は無機質な
ダーク管制端末ではなく、参考画像（Lauren Roth「Find Peace Within」）に倣う**やわらかい
ライトのパステル／ティール基調**の“穏やかな運用コンソール”とする（意匠の詳細は 0.6）。

同じ潮流・潮汐・警報・通航量が、アートでは「にじむ光」に、業務端末では「警報一覧」
「検潮所ステータス表」「通航量モニタ」に化ける——その**対比そのものが作品**。

実装者は上から順に、**1フェーズ完了ごとにビルド・ブラウザ確認・コミット**する。

## 0. 実装前の必読事項

### 0.1 うみ本体を壊さない・汚さない（最優先）

- 新サイトは**同一リポジトリ内の `console/` に独立した別ビルド**として作る。
  うみ本体（`index.html` ほか各作品ページ・`src/*.ts`）には手を入れない。
- three.js を入れるのは**この新サイトだけ**。うみの「ライブラリを足さない軽さが作品」
  という原則は据え置く。うみ本体の作品ナビ `nav.ts` の `PAGES` には
  **かいしょうを足さない**（うみの作品一覧には載せず、別世界として切り離す）。
  一方 `vite.config.ts` の rollup input には `console/index.html` を**追加してよい**
  （同一リポジトリで同居ビルドし `/umi/console/` に配信するため。フェーズ14で実施済み）。
  three 依存が混入するのは console バンドルのみで、うみ本体7ページのビルドは不変に保つ。
- 共有するのは**データ層 `src/api.ts` ただ一つ**。MSILクライアント（429リトライ・
  `arcgisQuery`・`toMsilTime` 等）は二重管理しない。console 側から相対 import で使う。
  `src/api.ts` に console 用の取得関数を**追記するのは可**（うみ側の既存export・挙動は変えない）。
  投影ロジック（`japanmap.ts` の Web メルカトル）は 2D 専用なので console では**流用しない**
  （3D は経度緯度→球面/平面メッシュ座標へ別途変換する。0.4 参照）。

### 0.2 エンドポイント・層番号は「推測で書かない」

PLAN3.md 0章と同じ鉄則。console が使う層も**設計時の想定**にすぎない。実装の最初に
必ず管理APIで機械的に確認してから書く:

- 一覧: `https://portal.msil.go.jp/mapi/apis?api-version=2022-04-01-preview`
- 操作と層enum: `https://portal.msil.go.jp/mapi/apis/{id}/operations?api-version=2022-04-01-preview`
- 層の自己記述: `https://api.msil.go.jp/{path}/v2/MapServer?f=json`（層名・フィールド名）

想定と実物が食い違ったら**実物に合わせる**。フィールド名は大小文字の揺れがあるため
`props.Foo ?? props.foo` の二段構えで拾う（`api.ts` の `fetchWaves` が前例）。
本計画で新規に叩く層はほぼ既存作品が実証済み（潮流 v3・潮汐 v3・航行警報 v2・波浪 v2・
船舶通航量 v2）。**まず既存作品のソースを読む**のが最短。

### 0.3 ネットワーク許可（サンドボックスで実装する場合）

Claude Code on the web 等の隔離環境では egress 許可が要る。Network access を Custom にし、
Allowed domains に以下を追加（「Also include default list…」はチェックのまま）:

```text
api.msil.go.jp
portal.msil.go.jp
maps.gsi.go.jp
```

three.js は npm から入るので、パッケージマネージャのデフォルト許可があれば追加不要。
ネットワークポリシーはセッション開始時に固定されるため、**設定後は新セッションで実装**。

### 0.4 三次元での座標・投影の方針

海しるは経度・緯度（EPSG:4326）で返る。three.js では以下のどちらか（フェーズ14で決める）:

- **A案 地球儀（推奨・signature 映え）**: 半径1の球に緯度経度→球面座標
  `(cosφ·cosλ, sinφ, cosφ·sinλ)` で載せる。日本近海に初期カメラを寄せる。
  レイヤ（点群・線・ヒートマップ）はすべて球面上に置く。OrbitControls で回転・ズーム。
- **B案 平面海図（情報密度重視）**: 日本近海 `JAPAN_BBOX` を等間隔平面メッシュに正規化。
  緯度経度→ `[0..1]×[0..1]` 線形。真上見下ろし＋わずかな傾き。業務端末感はこちらが強い。

まず **A案の地球儀**で土台を組む（絵の強さが保証される）。陸地シルエットは three.js 側で
低解像度の日本近海海岸線を用意するか、地理院タイルを球テクスチャに貼る（0.5）。

### 0.5 陸・海の背景

海しるは陸データを持たない。console では 3D 背景として以下のいずれか:

- 地理院 淡色地図タイル（`maps.gsi.go.jp`）を**球/平面のテクスチャ**として貼る（最も手軽・
  ライト基調にそのまま馴染む）。クレジットに「国土地理院」を明記（うみと同条件・利用規約）。
- もしくは自前の低ポリ日本近海海岸線を線で描く。**ライト背景では線は濃色ではなく
  やわらかいティール／スレート**にし、海面はごく淡い空→海のグラデーションで（0.6 の painterly 質感）。

### 0.6 デザイン言語（構造は業務システム／意匠は「ダーク管制センター（VTS/レーダー風）」）

> **改訂経緯**: 当初は参考画像「Find Peace Within」に倣うライト・パステル基調で実装したが、
> 中間色ティールの海の上でピン・粒子・海岸線・文字がすべて低コントラストになり
> **可読性が著しく低かった**。運用コンソールは読めることが最優先のため、実物のVTS・
> ECDIS・レーダー・ミッションコントロールに倣う**ダーク高コントラスト**へ全面変更した。

方針: **業務システムの骨格（パネル・データテーブル・レイヤ・インスペクタ・ステータスバー・
警報ティッカー）は保ちつつ、暗い海に明るいデータを浮かせる高コントラストにする**。
うみ（詩・ライト・明朝）の陰画として、こちらは**監視・ダーク**で対比を強める。テーマは**ダーク**。

- **配色（ダーク基調。トークンは `console/style.css` の `:root` に集約）**:
  - 背景 `--bg: #0a1826`（深い藍墨）〜 `--bg-2: #07111c`。パネル面 `--panel: #0c1826`／`--panel-2: #11202f`。
  - 本文 `--fg: #e4eef6`（ほぼ白・高コントラスト）、副次 `--fg-dim: #8aa0b4`。
  - 区切り `--line: rgba(110,150,180,0.20)` の細い明色線。角は少し丸める（`border-radius` 6–10px）。
- **アクセント＝機能色は3色（暗地で映える高輝度）**:
  - 正常／基調＝シアン `--ok: #5ee0d8`
  - 注意＝アンバー `--warn: #f0b64e`
  - 危険＝レッド `--alert: #ff6b5e`
- **「量」の表現は単一色相ランプ**（うみ `render.ts` の発想を継承）。ダーク地なので
  **暗いシアン→明るいシアン白**（暗い側＝小、明るい側＝大）で発光させる。虹色・多色は禁止。
  各レイヤの実際のランプ（潮流/波浪/潮位/通航量）は該当 `console/*.ts` の定数に持つ。
- **タイポグラフィ**: データ・数値・テーブルは可読性優先の**ヒューマニスト sans**
  （`ui-sans-serif, "Hiragino Sans", "Noto Sans JP", system-ui`）＋**数値は tabular-nums**。
  等幅（`ui-monospace`）は座標・電文・時刻・ログ**だけ**に限定。サイト名／ロゴの1箇所だけ手描き風スクリプト可。
- **質感（radar/console）**: 暗い背景に、データと海岸線・経緯線を明色で。3D海図は
  深い藍墨の海＋**大気の縁光（cyan の atmospheric haze）**で球の存在感を出す。LED や重要データに
  弱い発光（glow）を許す（ネオンにはしない）。海岸線は明るいシアングレーで日本をくっきり縁取る。
- **レイアウト（業務系の骨格は維持）**: ヘッダ（ロゴ＋JST時計＋ダミー職員ID＋接続ステータス）／
  左＝レイヤパネル／右＝インスペクタ（選択対象の生データを表で）／下＝ステータスバー＋警報ティッカー。
  配色・質感のみ上記のダークへ。情報密度は業務系のまま臆せず密に出す（時刻・座標・件数）。
- **クレジット（必須・うみと同じ法的義務）**: 「データ: 海上保安庁 海しるAPI（使用API名）
  ／（地理院タイル使用時）国土地理院／航海には使用しないでください」を常時表示。

### 0.7 再利用できる部品

- `src/api.ts` — MSILクライアント一式（**必ず経由**。console 用関数の追記可）
- 各既存作品の**データ取得ロジック**は console の各レイヤの下敷き:
  - 潮流 → `fetchCurrents` / `fetchAreas`（`main.ts`）
  - 潮汐 → `fetchTideStations` / `fetchTideDay`（`tide.ts`）
  - 航行警報 → `arcgisQuery("navigational-warnings/v2", …)` とページング・電文整形（`koe.ts`）
  - 通航量 → `michi.ts`、波浪 → `fetchWaves`（`api.ts`）
- **three.js は新規**。粒子・ラインは three.js の `Points` / `Line` / `InstancedMesh` で作り直す
  （うみの Canvas 2D 描画コードは移植しない。データ構造だけ共有する）。

## フェーズ15: 警報コンソール — 航行警報（v2）★最小コスト・物語性最大

**ゴール**: こえと同じ航行警報データを、業務端末の**アラームコンソール**として。
右に一覧テーブル、地図（球）上に点滅ピン、種別フィルタ、選択でインスペクタに電文全文。
うみの「こえ」の陰画。実装コスト最小・物語性最大なので最初のデータレイヤに据える。

**データ**: `koe.ts` の `fetchLayer`（`navigational-warnings/v2` 層1=点・層3=面、
ページング・`cleanDescription`・`detectCategory`）を **console 用に移植/共有**。
`api.ts` に `fetchNavWarnings()` として切り出し、こえ・console 双方から使えると理想
（ただしこえの挙動は不変に保つ）。

**ビジュアル/UI仕様**:
- 3D: 各警報を球面上の `InstancedMesh`（小円錐/球）で。種別色（正常ティール/注意ソフトアンバー/
  危険テラコッタの3色枠内に `koe.ts` の5カテゴリを写像。0.6 の機能色）。危険系はやわらかく明滅
  （ネオンの点滅ではなく、にじむようなゆっくりした呼吸）。
- 右インスペクタ: 警報番号・海域・発令時刻・電文全文（等幅・スクロール）。
- 左パネル: 「航行警報」レイヤのトグルと件数バッジ。種別フィルタ（こえのチップを表形式に）。
- 下ステータスバー右端に**警報ティッカー**（新しい番号順に電文見出しを流す）。
- ホバーで球上のピンをハイライト＋テーブル行を連動スクロール（双方向）。

**確認**: 実際の有効警報件数が出て、地図ピンとテーブルが連動。種別フィルタが効く。
インスペクタに電文全文。接続ステータスLEDが緑。コンソールエラーなし。

## フェーズ16: 海況レイヤ — 潮流（3D粒子）＋波浪（ヒートマップ）

**ゴール**: 自然の海況を切替可能なレイヤとして重ねる。潮流を球面上の three.js 粒子で流し、
波浪を有義波高のヒートマップ（点群 or 球面グリッド）で。レイヤトグルと凡例が本格稼働。

**データ**:
- 潮流: `fetchAreas` / `fetchCurrents`（`main.ts`）。海域を選ぶUI（左パネルのセレクト）。
  時刻は `config.ts` の `TIME_SPAN_*` を流用し、下ステータスバーに時刻スクラバを置く。
- 波浪: `fetchWaves(JAPAN_BBOX)`（`api.ts`）。有義波高を単一色相ランプで。

**ビジュアル/UI仕様**:
- 潮流粒子: three.js `Points` を球面接ベクトルに沿って移流（うみの VectorField の IDW は
  概念だけ借り、3D用に `console/field3d.ts` として実装。過度な共通化はしない）。
  色ランプは流速、`max` は既存 `main.ts` 準拠。
- 波浪: 球面グリッドの頂点色 or 半透明点群で高波域を明るく。凡例「0 →→ 最大波高 m」。
- 左パネルのレイヤトグルで警報/潮流/波浪を独立ON/OFF。凡例はアクティブレイヤに追従。
- 右インスペクタ: ホバー地点の `流速 1.8 kt・流向 SW` / `有義波高 2.4 m` を数値表示。

**確認**: 潮流粒子が実流向で流れ、時刻スクラバで変化。波浪の高波域が地理的に妥当
（外洋・季節に整合）。レイヤトグルが独立に効く。性能: 粒子は間引いて 60fps 目安。
コンソールエラーなし。

## フェーズ17: 検潮所監視盤 — 潮汐（v3）

**ゴール**: 全国検潮所を**監視盤**として。地図上の局マーカー＋右に一覧表（局名・現在潮位・
満干）＋選択で潮位曲線パネル。しおどきの「呼吸する地図」の陰画＝運用ダッシュボード。

**データ**: `fetchTideStations` / `fetchTideDay`（`tide.ts`）。当日分。現在時刻の潮位を
1分値配列から取り出して「現在値」に。

**ビジュアル/UI仕様**:
- 3D: 各局を球面マーカー。現在潮位を単一色相ランプ or マーカー高さ（3Dならではの押し出し）。
- 右インスペクタ: 選択局の**潮位曲線**（当日1440点）を、やわらかい淡ティールの折れ線＋
  低コントラストのグリッドで（0.6 の質感に沿う）。満潮・干潮時刻。
- 一覧テーブル: 局名・現在潮位[cm]・トレンド（↑/↓）。列ソート可（業務表らしく）。
- 全局同時取得は件数が多い → 初回は代表局、選択時に個別取得の遅延ロード（負荷配慮）。

**確認**: 主要検潮所が地図と表に出て、選択で潮位曲線が描ける。現在潮位が妥当な範囲。
遅延ロードでAPIを叩きすぎない。コンソールエラーなし。

## フェーズ18: 通航量モニタ＋仕上げ — 船舶通航量（v2）

**ゴール**: AIS 船舶通航量を**トラフィックモニタ**レイヤに。主要港・海峡が過密として灯る。
併せて全体の仕上げ（レスポンシブ・空状態・エラー処理・クレジット・READMEリンク）。

**データ**: `michi.ts` の取得（`vessel-traffic/v2` 相当・ページング・対数スケール）を流用。
メッシュ集計値（通航隻数）。

**ビジュアル/UI仕様**:
- 3D: メッシュ重心を球面点群で、通航量の**対数**を単一色相ランプに（湾内と外洋の桁差対策）。
  静的1回描画（毎フレーム再計算しない）。
- 左パネルにレイヤ追加。右インスペクタで `通航 1,240 隻/月`。凡例に対数目盛注記。
- **仕上げ**:
  - モバイル幅（<720px）で最低限崩れない（パネルは折り畳み/ドロワー化）。
  - 各レイヤの空状態（0件）を業務的に明示（「該当データなし」）。API失敗時は
    ステータスLEDを赤にしトースト表示（うみは沈黙だったが業務端末は明示する）。
  - クレジットに使用API名を全部列挙（航行警報・潮流推算・潮汐推算・波浪解析・船舶通航量
    ＋地理院タイル＋「航海には使用しないでください」）。
  - うみ本体の README または index にさりげない相互リンク（任意・うみの意匠を乱さない範囲で）。

**確認**: 主要港・海峡が明るい帯として浮かぶ。全レイヤが独立トグルで共存。モバイルで最低限。
`npm run build` 通過・うみ7ページ回帰なし。コンソールエラーなし。
