## 0. 実装前の必読事項（第3期で新しく効いてくる制約）

### エンドポイント・層番号は「推測で書かない」

この計画に書く各APIの `path` と層番号は**設計時の想定**であり、確定値ではない。
実装の最初に、必ずポータルの管理APIで機械的に確認してから書く（PLAN.md 0章に既出）:

- 一覧: `https://portal.msil.go.jp/mapi/apis?api-version=2022-04-01-preview`
- 操作と層enum: `https://portal.msil.go.jp/mapi/apis/{id}/operations?api-version=2022-04-01-preview`
- データ層の自己記述: `https://api.msil.go.jp/{path}/v2/MapServer?f=json`（層名・フィールド名が出る）

想定と実物が食い違ったら**実物に合わせる**。フィールド名は大小文字の揺れがあるため、
`fetchWaves`（`src/api.ts`）と同じく `props.Foo ?? props.foo` の二段構えで拾う。

### ネットワーク許可（Claude Code on the web 等のサンドボックスで実装する場合）

海しるはローカル/本番（GitHub Pages）のブラウザからは直接叩けるが、隔離環境では
egress許可が要る。環境の Network access を **Custom** にし、Allowed domains に:

```text
api.msil.go.jp
portal.msil.go.jp
maps.gsi.go.jp
```

を追加（「Also include default list of common package managers」はチェックのまま）。
ネットワークポリシーはセッション開始時に固定されるため、**設定後は新しいセッションで実装する**。

### 守るもの（PLAN.md デザイン言語の再掲・厳守）

- 背景 `#04070f`、インク `rgba(226,238,248,…)`、アクセント `#7fe3e0`。トークンは `src/style.css` の `:root`
- 量の表現は**単一色相のシーケンシャルランプ**（`render.ts` の `speedColor` を流用）。虹色・多色禁止。
  カテゴリ色分けは最大3色まで。新しい色を足すのは意味がある1色に限る（こえの琥珀 `#e8c468` が前例）
- 明朝体・広いletter-spacing・四隅の小さなHUDのみ。枠/カード/影なし
- 左下クレジット（海しる＋使うAPI名、地理院タイル使用時は「国土地理院」、「航海には使用しないでください」）
- 各ページ末尾で `mountNav("<id>")` を呼ぶ。新ページは `src/nav.ts` の `PAGES` と
  `vite.config.ts` の rollup input に追加する
- **フレームワーク・外部ライブラリ・地図/3Dライブラリは足さない**（この軽さが作品の一部）

### 再利用できる部品（第3期の土台）

- `JapanMap`（`src/japanmap.ts`）— 日本全域のWebメルカトル投影＋陸シルエット。
  `toScreen(lon,lat)` / `toLonLat(px,py)` / `drawBase()`。**面/点の全域作品はこれ一択**
- `VectorField` と `lerpLookup`（`src/field.ts`）— 散在点からIDWでベクトル場を作り、
  2時刻を線形補間。粒子を流す作品（ながれ）の心臓部。**かぜはこれをそのまま使う**
- `render.ts` — `speedColor(v, max)` の単一色相ランプ、粒子トレイル描画
- `landmask.ts` — 地理院タイルの海色判定（`JapanMap` 経由で使うのが基本）
- `api.ts` — `msilFetch`（429リトライ内蔵・**必ず経由**）、`arcgisQuery(path, layer, bbox, extra?)`、
  `toMsilTime` / `toMsilDate`（JST変換）

## フェーズ8: みなも — 海の体温（海面水温）★signature

**ゴール**: 日本全域の海面水温を、群青（冷）→シアン→白（暖）の一枚の温度図に。
黒潮・対馬暖流の**暖かい舌**が熱として浮かび上がり、くろしおページと対になる代表作。

**データ**:
- 想定 `sea-surface-temperature/v2`（**要確認**。海面水温解析。管理APIで path と層・フィールド名を確定）。
  フィールドは水温[℃]（`Temperature` / `SST` 等、実名は自己記述で確認）
- 全域 `JAPAN_BBOX`。1000件上限に当たるはず → `exceededTransferLimit` を見て `resultOffset` でページング
  （`api.ts` にページング付き汎用取得ヘルパー `arcgisQueryAll(path, layer, bbox, extra?)` を新設し、
  等深線・沈船の取得もいずれこれに寄せる）
- 点群（散在サンプル）で返る想定。ラスタ(PNG)しか無い層なら**この作品は点描を諦めず**、
  格子点をGeoJSONで取れる層（解析格子）を優先的に探すこと

**ビジュアル仕様**:
- `JapanMap` で全域投影＋淡い陸シルエット。`drawBase()` 済みの上に描く
- 各水温サンプルを `speedColor(t, MAX)` の色で、なみと同様「にじみ＋芯」の二重円で点描
  （`ctx.globalCompositeOperation = "lighter"`）。色スケールは**当日の実データの
  p5〜p95で正規化**（絶対0〜30℃固定だと夏冬で沈む。しおどきの正規化と同じ発想）
- 密度が粗ければ `VectorField` の IDW（`lookup`）を**スカラー版に一般化**して面で塗ってもよいが、
  まずは点描で軽く出す。重くなるなら格子を間引く
- 左上タイトル「みなも」/ "Sea Surface Temperature"、下に解析時刻と水温レンジ（例 `18.4–27.1 ℃`）
- ホバー: 最近傍サンプルの `水温 24.3 ℃` を `#tooltip` で表示
- 凡例: なみと同じ横グラデ帯。ラベルは「冷 ←→ 暖」＋両端に実温度

**確認**: 黒潮流路に沿って高温域が帯状に見え、北ほど低温。コンソールエラーなし。
ページング時も全域が埋まる。スクショで温度勾配が地理的に妥当（南岸沖が暖かい）こと。

## フェーズ9: かぜ — 海の上を吹くもの（海上風）★最小コスト

**ゴール**: ながれ（潮流）の粒子機構をそのまま流用し、海上風の風向・風速で数千の粒子を
泳がせる。水（ながれ）と空（かぜ）の対。実装コストが最も低く、絵の強さは保証されている。

**データ**:
- 想定 `sea-surface-wind/v2` あるいは気象GPV系（**要確認**。海上風の格子データ）。
  各点に風向[°]（北=0、時計回り、**吹いて来る方向**か**行く方向**かをレスポンスで判定）と
  風速[m/s]。海しるは風向を「風が吹いてくる方位」で持つことが多いので、粒子の進行方向は
  **その逆**にする（実装時に矢羽の向きで妥当性を目視確認）
- 取得は `arcgisQuery`。格子なので1リクエストで全域が返るはず。返らなければ bbox 分割

**再利用（ほぼ書かない）**:
- `field.ts` の `VectorField` は `CurrentSample{lon,lat,dir,kt}` を受ける。
  **風のサンプルを `{lon,lat,dir, kt: 風速}` に写像**して同じクラスに流し込む
  （kt/m/s の単位は色スケールの `max` を風用に変えるだけ。物理量としては変換不要）。
  `main.ts` の粒子ループ・`lerpLookup` による時刻補間はそのまま
- `main.ts` を関数に切り出せていない場合、**新規 `kaze.ts` は `main.ts` を薄く複製**して
  データ源だけ差し替える（過度な共通化はしない。ながれを壊さないことを優先）

**ビジュアル仕様**:
- ながれと同じ粒子トレイル。色ランプは同じ `speedColor` だが `max` は風速用（例 20 m/s）
- 時刻スライダーは `config.ts` の `TIME_SPAN_*` を流用（海上風の予報が複数時刻取れる場合）。
  取れなければ最新1時刻のみ（スコープを膨らませない）
- 左上「かぜ」/ "Sea Surface Wind"、解析/予報時刻。ホバーで `風速 12 m/s ・ 北西` 等
  （8方位の和名変換を小関数で）
- 台風接近時に画面が渦を巻く — それがデータの見どころ

**確認**: 粒子が現実の卓越風向（冬の北西季節風など季節に応じた向き）で流れる。
ながれページが従来どおり動く（回帰させない）。コンソールエラーなし。

## フェーズ10: みち — 人の海（船舶通航量 / AIS）

**ゴール**: 海上保安庁がAISで集めた**船舶通航量（月間）**を、光る「海の道」として描く。
自然の海（潮流・波）に対する**人の海**。東京湾・伊勢湾・瀬戸内・関門・津軽が動脈のように灯る。

**データ**:
- 想定 `vessel-traffic/v2`（月間通航量。**要確認**。メッシュ集計値=通航隻数のポリゴン/点）。
  値フィールド（隻数）を自己記述で確認。全国メッシュは件数が多い → **必ずページング**
- 最新1か月分のみ。属性に対象年月があれば表示に使う

**ビジュアル仕様**:
- `JapanMap` 全域投影。各メッシュを通航量で `speedColor`（暗群青=閑散 → 白=過密）。
  メッシュ塗り or 重心点描。**対数スケール**推奨（湾内と外洋で桁が違うため線形だと湾しか光らない）
- 過密ほど明るく・わずかに大きく。`lighter` 合成で航路が線としてつながって見える
- 左上「みち」/ "Vessel Traffic"、対象年月。ホバーで `通航 1,240 隻/月`
- 凡例: 「閑散 ←→ 過密」＋対数目盛の注記

**確認**: 主要港・海峡が明るい帯として浮かぶ。外洋は暗く沈む。ページングで全国が埋まる。
コンソールエラーなし。**性能**: メッシュが多いので描画は静的1回（毎フレーム再描画しない）。

## フェーズ11: こおり — 凍る海（海氷）※季節もの・任意

**ゴール**: オホーツク海・北海道沖の**海氷密接度**を白の点描で。冬にだけ現れ、
夏は「凍っていない」ことをデータが静かに語る。ねむり同様、演出は最小。

**データ**: 想定 `sea-ice/v2`（**要確認**。密接度[%] または区分）。bbox は北日本
`[139, 43, 149, 46]` に絞ってよい（全域だと空白が広い）。

**ビジュアル仕様**:
- 北日本に寄せた投影（`JapanMap` に bbox を渡す）。密接度で白の濃淡（`speedColor` の高温端＝白を流用）
- **オフシーズンの正直さ**: データ0件なら「いまオホーツクに海氷はありません（結氷期: 1〜3月）」と
  メタ行に出す。空を空として見せる（ねむりの精神）
- 左上「こおり」/ "Sea Ice"、密接度レンジ。ホバーで `密接度 70 %`

**確認**: 冬の日付で北海道沿岸に白域が出る。夏はオフシーズン表示。コンソールエラーなし。

## フェーズ12: すじ — 海の底の神経（海底ケーブル）※任意

**ゴール**: 日本と世界をつなぐ海底ケーブル経路を、海底地形の上に微光の線で。静的データ。
ねむり同様「眺めるだけ」の静かなページ（PLAN.md でも次期候補として名指しされている）。

**データ**: 想定 `submarine-cable/v2`（**要確認**。LineString、属性にケーブル名があれば使う）。
全域 `JAPAN_BBOX`。線が多ければページング。

**ビジュアル仕様**:
- `JapanMap` 全域＋陸シルエット。ケーブルを `rgba(127,227,224,0.18)` 程度の細線で敷き、
  上を微光の点がゆっくり流れる（くろしおの弧長パラメータ移動を流用）。多色にしない
- 左上「すじ」/ "Submarine Cables"。名前があればホバーで表示、無ければ点数のみメタ行に
- くろしおの流軸移動ロジック（`kuroshio.ts`）が最も近い前例。まずそれを読む

**確認**: 日本近海のケーブル網が視認でき、陸揚げ地点付近に集まる。コンソールエラーなし。

## フェーズ13: さかい — 見えない境界（領海・EEZ）※任意

**ゴール**: 領海・排他的経済水域の境界を、日本を囲むかすかな発光線として。概念的で静謐な一枚。

**データ**: 想定 `territorial-sea/v2` / EEZ相当（**要確認**。境界ポリゴン/ライン）。全域。

**ビジュアル仕様**:
- `JapanMap` 全域＋陸シルエット。境界線を1〜2色（領海＝アクセント、EEZ＝より淡いインク）
  の細線でなぞる。塗りつぶさず線だけ。ゆっくり明滅させても可（主張しすぎない）
- 左上「さかい」/ "Territorial Waters & EEZ"。ホバー・クリックなし（眺めるだけ）

**確認**: 日本を大きく囲むEEZ外縁と、沿岸の領海線が見分けられる。コンソールエラーなし。
